# How Proof of Origin Works

This document explains the system in plain terms. No cryptography background needed — just a willingness to follow the logic.

---

## The Core Problem

When you see an image online, you have no way to know:
- Was it generated by AI or taken by a human?
- If it was AI-generated, which system made it?
- Is this the original file, or a screenshot / crop / edited version?

Metadata gets stripped. Screenshots lose their history. A single re-save cuts the connection to the source. Proof of Origin is a system that makes forgetting impossible — the image carries its own proof of origin, and that proof lives permanently on a public blockchain that anyone can query.

---

## The Big Picture

When a user generates an image through this app, four things happen at once:

1. **The image gets invisibly marked** — three independent watermarks are woven into the pixels
2. **A fingerprint is computed** — mathematical summaries (hashes) uniquely describe this exact image
3. **A cryptographic proof is generated** — a zero-knowledge proof that ties the fingerprint to the watermark
4. **Everything gets anchored** — the proof goes to a blockchain, the full record goes to Arweave (a permanent decentralized archive)

When someone later uploads *any version* of that image — the original, a screenshot, a JPEG shared three times — the app reads the watermarks back out, checks the blockchain, and tells you exactly what it knows.

---

## Layer 1: Three Watermarks

Think of the watermarks as three different ways of signing your name on the same document — each in a different ink, each resistant to different kinds of damage.

### Watermark A — The Frequency Fingerprint (strongest)

Images can be described mathematically as patterns of frequency — like how music can be broken into bass, mid, and treble. This watermark hides a 16-character ID in the **low-frequency structural patterns** of the image (specifically, the wavelet sub-bands that encode textures and edges).

Why is this strong? Because when an AI model regenerates an image — taking your output and running it through img2img or a similar process — it preserves the broad structural layout: the shapes, textures, and lighting. The watermark lives in exactly those structures, so it survives even aggressive AI regeneration.

The verification step extracts the hidden ID and checks how closely it correlates with the expected value. A score ≥ 70% counts as a match.

### Watermark B — The Neural Watermark (everyday resilience)

This uses a machine-learning model called TrustMark (developed by Adobe/CAI) that encodes the same 16-character ID into the image's pixels in a way that survives JPEG compression, resizing, and format conversion.

Where watermark A lives in frequency space, this one lives in the actual pixel values — but encoded so subtly that it's invisible to the human eye. It breaks only under pixel-level destruction (extreme cropping that removes the center, or deliberate pixel-scrambling attacks).

### Watermark C — The Metadata Label (easiest to read)

A plain JSON record — watermark ID, timestamp, short ID — is embedded directly in the PNG file's metadata section (iTXt chunk). This is the simplest and least robust: any JPEG re-encode strips it. But it's also the easiest to recover, and it integrates with the C2PA (Content Credentials) industry standard used by Adobe, cameras, and major platforms.

### Why Three?

Each watermark survives different attacks. An attacker would need to destroy all three simultaneously — a very high bar. In practice, ordinary operations (screenshot, crop, re-share) leave at least one intact.

## Layer 2: Hashes

A **hash** is a mathematical fingerprint. Feed any file into a hash function and you get a fixed-length string — like `a3f8c2...` — that uniquely represents that file. Change a single pixel and the hash changes completely. The same file always produces the same hash.

This system computes two kinds of hashes:

### The Exact Hash (SHA-256)

A cryptographic hash of the complete watermarked PNG file. If you upload the exact same bytes you downloaded, the hashes will match. If you re-save as JPEG, the hash changes — that's expected, and the system will classify you as "transformed" rather than "original".

### The Perceptual Hash (DINOv2)

A hash that captures *visual meaning* rather than exact bytes. It uses Facebook's DINOv2, a vision AI model that was trained to understand what images look like to humans. The system feeds the image into DINOv2 and gets a 384-dimensional embedding — essentially a 384-number description of the image's visual content. It then picks 64 of those numbers and converts them to a 64-bit integer.

Two images that "look the same" to a human — even after screenshots, crops, or re-encoding — will produce similar DINOv2 hashes. The **Hamming distance** (number of differing bits) between two such hashes will be small, typically under 15. Two completely different images will have a distance near 32 (random chance).

This hash is what allows the system to recognize a heavily compressed screenshot of an AI image as "transformed" rather than "unrecognised."

## Layer 3: Zero-Knowledge Proofs

This is the most technically unusual part, so here is a careful explanation.

### The Problem a ZK Proof Solves

The blockchain stores hashes. Anyone can see those hashes. But the hashes alone don't prove that the person who registered the image *actually had the image at registration time*. They could have registered hashes speculatively, then tried to find an image that matched.

A zero-knowledge proof solves this. It lets the system prove: **"I know the secret values (the short ID, the exact image hash, and the perceptual hash) that correspond to this registration — without revealing those values."**

### What the Proof Looks Like in Practice

The circuit takes 384 numbers as input:
- 64 bits encoding the short ID
- 256 bits encoding the image's SHA-256 hash
- 64 bits encoding the perceptual (DINOv2) hash

It runs them through a minimal neural network (a single ReLU activation layer — technically the simplest non-trivial computation). Then it generates a cryptographic proof that this computation was done correctly using those specific inputs.

The proof has two important properties:
1. **Soundness**: It's computationally impossible to generate a valid proof without knowing the actual input values
2. **Zero-knowledge**: The proof doesn't reveal the input values — only that they exist and were used

### Where the Proof Goes

The proof is submitted to a Solidity smart contract on the Base Sepolia blockchain. That contract calls a verifier contract (automatically generated by the ezkl tool from the circuit definition) which checks the math. If the proof is valid, the contract stores `proofVerifiedOnChain = true`. If it's invalid, the entire transaction reverts — the registration is rejected.

This means the blockchain itself verified the proof, not just stored a hash. Anyone can check that the `proofVerifiedOnChain` flag is true without trusting this server at all.

---

## Layer 4: The Blockchain Record

The smart contract on Base Sepolia stores:

```
watermarkId   → keccak256 of the short ID (a one-way derived key)
imageHash     → keccak256 of the image's SHA-256 string
perceptualHash → 64-bit DINOv2 hash (allows derivative checks)
proofVerifiedOnChain → true/false
timestamp     → when the block was mined
creator       → the wallet address that registered the image
```

Registration is split into two phases deliberately:

**Phase 1 — Placeholder**: A fast, cheap transaction (~45k gas) that reserves the slot immediately after image generation. This prevents a crash from causing the image to be watermarked but never registered.

**Phase 2 — Complete Registration**: A second transaction that fills in the actual hashes and the ZK proof. This is safe to retry — a second call reverts with "Already registered."

The split means the only unrecoverable failure is a crash *before* the placeholder is created — and by that point, the image hasn't even been watermarked yet.

---

## Layer 5: Arweave Permanent Storage

The blockchain record is permanent but expensive to store text in and limited in size. Arweave is a decentralized permanent storage network. The system uses Irys (a bundling service on top of Arweave) to upload the full provenance record:

```json
{
  "watermark_id": "...",
  "short_id": "...",
  "sha256": "...",
  "phash": 12345678901234,
  "prompt": "a cat on the moon",
  "model": "flux-2-pro",
  "created_at": 1712345678,
  "tx_hash": "0x..."
}
```

This record is human-readable and permanently accessible at `https://gateway.irys.xyz/{transaction_id}`. Even if this server disappears forever, the record remains.

## The Verification Flow

When someone uploads an image to verify:

**Step 1 — Extract the short ID**

The system tries three recovery paths in order:
1. Decode TrustMark neural watermark from the pixels
2. Read the PNG metadata chunk
3. Scan the local database for images whose DINOv2 hash is close enough (Hamming distance ≤ 15)

If all three fail, the image was never registered here, or every recovery signal was destroyed.

**Step 2 — Look up the blockchain**

Using the recovered short ID, the system queries the contract:
```
lookup(keccak256(short_id)) → (imageHash, perceptualHash, proofVerifiedOnChain, timestamp, creator)
```
This is a free, read-only call — no gas, no wallet, no trust in this server.

**Step 3 — Compare**

- Hash the uploaded file with SHA-256 → compare to `imageHash` on-chain
  - Match → **"original"** (exact bytes)
  - No match → compute Hamming distance between uploaded and stored DINOv2 hashes
    - Distance < 15 → **"transformed"** (screenshot, crop, JPEG re-encode, AI regeneration)
    - Distance ≥ 15 → **"none"** (unrecognised — too different)

**Step 4 — Verify the ZK proof locally**

If a proof was generated at registration time, it's stored server-side and can be re-verified locally without a blockchain call. This gives an independent cryptographic confirmation.

## What "Trustless" Means Here

Most of the verification steps don't require trusting this server:

| What you're checking | What you need |
|---|---|
| Recover short ID from watermarked image | The image file + open-source TrustMark library |
| On-chain registration record | A Base Sepolia RPC endpoint |
| Derivative match (pHash similarity) | A Base Sepolia RPC — the contract has a `verifyDerivative()` function |
| ZK proof validity | A Base Sepolia RPC — the verifier contract checks it on-chain |

The one thing that does require the backend: recovering a short ID from an image where *all* watermarks have been destroyed, using DINOv2 pHash proximity search against the database. That scan depends on this server's SQLite database and is the only non-decentralized part of the system.

## The Full Journey of One Image

```
User types: "a red cube on a white background"
         ↓
AI model (Flux, DALL-E, etc.) generates a PNG
         ↓
Three watermarks are embedded into the pixel data
         ↓
SHA-256 and DINOv2 hash are computed
         ↓
Image is returned to the user immediately
         ↓
  (background, takes 10–60 seconds)
         ↓
ZK proof is generated from (short_id, sha256, phash)
         ↓
createPlaceholder() → Base Sepolia blockchain
         ↓
completeRegistration(sha256, phash, zkProof) → Base Sepolia blockchain
  → Verifier contract checks the ZK proof math
  → Record stored permanently: proofVerifiedOnChain = true
         ↓
Full provenance JSON uploaded to Arweave via Irys
         ↓
Anyone, anywhere, can now upload any version of that image
and get: who made it, when, with which model,
whether it's the original or a derivative,
and a cryptographic proof that the registration was honest.
```

## Stack Summary

| Component | Technology | Role |
|---|---|---|
| Image generation | OpenRouter (Flux, DALL-E, Gemini) | Create the image |
| Semantic watermark | DWT-DCT-SVD (wavelet + SVD) | Survives AI regeneration |
| Neural watermark | TrustMark (Adobe/CAI) | Survives JPEG, resize, re-encode |
| Perceptual hash | DINOv2 ViT-S/14 | Detects visual similarity across transforms |
| ZK proof | ezkl (ONNX → Halo2 → Solidity verifier) | Cryptographic binding, verified on-chain |
| Blockchain | Base Sepolia (Ethereum L2) | Permanent, public, tamper-proof record |
| Permanent storage | Arweave via Irys | Full provenance survives server shutdown |
| Backend | FastAPI + Python | Orchestrates all of the above |
| Frontend | React + TypeScript | Generate and verify UI |
